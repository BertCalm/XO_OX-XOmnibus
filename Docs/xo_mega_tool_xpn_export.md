## 1. Overview

XOmnibus exports presets as MPC-compatible expansion packs (`.xpn`) for use in Akai MPC software and hardware. The export pipeline renders synth presets to multi-sampled WAV files, builds XPM keygroup programs, and packages everything into installable expansions.

**Key principle:** The coupling DSP cannot be exported to MPC — only rendered audio. Multi-engine presets are rendered through the full coupling chain, then captured as WAVs.

**Existing tooling:**
- `tools/xpn_export.py` — XPN packager (from XOdyssey, needs update)
- `tools/fix_xobese_xpms.py` — XPM fixer (documents the 3 critical rules)
- Working reference: DX-TX v2 expansion at `~/Library/Application Support/Akai/MPC/Expansions/com.amazound.dxtxv2/`

## 2. The 3 Critical XPM Rules

Discovered by comparing broken XObese XPMs against working DX-TX v2 expansion:

| Field | Must Be | Why | Previous (Broken) |
|-------|---------|-----|-------------------|
| `KeyTrack` | `True` | Samples transpose across the keygroup zone | Was `False` |
| `RootNote` | `0` | MPC auto-detect convention — MPC determines root from filename/position | Was `MIDI+1` |
| Empty layer `VelStart` | `0` | Prevents ghost-triggering layers that have no sample assigned | Was `1` |

**These rules apply to ALL XPM files generated by XOmnibus. No exceptions.**

## 4. Rendering Pipeline

### 4.1 Step-by-Step

```
Step 1: Load .xometa preset
        ↓ Activate engine(s), coupling, FX
Step 2: Configure render settings
        ↓ Sample rate, bit depth, note range, velocity layers, render length
Step 3: For each note in range:
        ↓ Send MIDI noteOn (velocity per layer)
        ↓ Render N seconds to float buffer
        ↓ Capture noteOff + tail (release + FX decay)
        ↓ Trim silence from end (threshold: -60dB)
        ↓ Normalize to -0.3dB ceiling
        ↓ Save as WAV
Step 4: Build XPM keygroup program
        ↓ One instrument per MIDI note zone
        ↓ Apply the 3 critical rules (KeyTrack=True, RootNote=0, empty VelStart=0)
Step 5: Build XPN manifest
        ↓ Package metadata, WAV list, XPM list
Step 6: Package into .xpn
        ↓ ZIP archive with correct directory structure
```

### 4.2 Render Settings

| Setting | Default | Options | Notes |
|---------|---------|---------|-------|
| Sample rate | 44100 Hz | 44100, 48000 | Match MPC project rate |
| Bit depth | 24-bit | 16, 24 | 24-bit recommended |
| Note range | C1-C6 (every minor 3rd) | Configurable | 21 notes default |
| Note selection | Every minor 3rd | Chromatic, every 3rd, every 5th, octaves only | Tradeoff: quality vs size |
| Velocity layers | 1 | 1, 2, 3 | More layers = more WAVs but better expression |
| Render length | 4 seconds | 2-10 seconds | Includes attack + sustain + release |
| Tail capture | 2 seconds | 1-5 seconds | FX tail after noteOff |
| Normalization | -0.3 dB | Off, -0.3, -1.0, -3.0 | Peak normalization |

### 4.3 Multi-Engine Preset Rendering

For presets with 2+ engines and coupling:

1. All engines are activated with their coupling routes
2. MIDI goes to the primary engine (first in `engines` array)
3. Coupling runs in real-time during render (the coupled interaction IS the sound)
4. The master output (post-coupling, post-FX) is captured
5. Result: a single WAV per note that contains the full coupled sound

**This means the coupling character is "baked in" — the MPC user gets the coupled sound but can't adjust coupling. This is acceptable because the rendered sound IS the preset's identity.**

### 4.4 WAV Naming Convention

```
{PRESET_NAME}__{NOTE}__{VELOCITY_LAYER}.WAV
```

Examples:
- `Velvet_Cosmos__C2__v1.WAV`
- `Velvet_Cosmos__Eb2__v1.WAV`
- `Velvet_Cosmos__C2__v2.WAV` (velocity layer 2, if multi-layer)

Rules:
- Double underscore `__` as separator (single underscore allowed in preset name)
- Note names: C, Db, D, Eb, E, F, Gb, G, Ab, A, Bb, B (flats, not sharps)
- Velocity layers: v1 (soft), v2 (medium), v3 (hard)
- Spaces in preset names replaced with underscores
- Max filename length: 60 characters

## 5. Drum Kit Export (ONSET Module)

ONSET presets export differently — they're drum kits, not melodic instruments:

### 5.1 Kit Export Format

- 8 voices → 8 pads in one XPM drum program
- Each voice rendered at its default pitch (from XOnset voice assignments)
- Pad mapping follows General MIDI drum map positions

| Voice | Name | Default Note | Pad |
|-------|------|-------------|-----|
| V1 | Kick | C1 (36) | Pad 1 |
| V2 | Snare | D1 (38) | Pad 2 |
| V3 | HH Closed | F#1 (42) | Pad 3 |
| V4 | HH Open | A#1 (46) | Pad 4 |
| V5 | Clap | D#1 (39) | Pad 5 |
| V6 | Tom | A1 (45) | Pad 6 |
| V7 | Perc A | C#1 (37) | Pad 7 |
| V8 | Perc B | G#1 (44) | Pad 8 |

### 5.2 MACHINE Variant Rendering

For ONSET kits, optionally render 3 variants of each kit:

| Variant | MACHINE Macro | Character |
|---------|--------------|-----------|
| Circuit | 0.0 | Pure analog (808/909 character) |
| Hybrid | 0.5 | Blended (unique to XOnset) |
| Algorithm | 1.0 | Digital (FM/Modal/K-S/PD) |

This produces 3 XPM programs per kit preset:
- `808_Reborn_Circuit.xpm`
- `808_Reborn_Hybrid.xpm`
- `808_Reborn_Algorithm.xpm`

### 5.3 Velocity Layers for Drums

Render each voice at 3 velocities to capture the `snap` parameter response:

| Layer | Velocity | Snap Behavior |
|-------|----------|--------------|
| v1 (soft) | 64 | Gentle transient |
| v2 (medium) | 100 | Moderate snap |
| v3 (hard) | 127 | Maximum transient sharpness |

## 8. Batch Export & Caching

### 8.1 WAV Cache

Rendered WAVs are cached to avoid re-rendering when the same preset appears in multiple bundles:

```
~/Library/Application Support/XO_OX/XOmnibus/ExportCache/
├── Velvet_Cosmos/
│   ├── Velvet_Cosmos__C1__v1.WAV
│   ├── Velvet_Cosmos__Eb1__v1.WAV
│   └── ...
├── Classic_Dub_Bass/
│   └── ...
└── cache_manifest.json   (preset name → hash of .xometa parameters)
```

- Cache key: SHA-256 of the `.xometa` parameter block
- If preset changes, cache is invalidated for that preset
- User can clear cache manually (Settings → Export → Clear Render Cache)

### 8.2 Batch Queue

When exporting multiple bundles:
1. Build unique preset set across all bundles
2. Render each unique preset once (to cache)
3. Package each bundle from cached WAVs
4. Report: "Rendered 200 presets, packaged into 6 bundles"

## 11. Size Optimization

| Strategy | Savings | Trade-off |
|----------|---------|-----------|
| Every minor 3rd instead of chromatic | 75% fewer WAVs | Slight pitch-shift artifacts at ±1.5st |
| Single velocity layer | 66% fewer WAVs vs 3-layer | Less expressive on MPC pads |
| 16-bit instead of 24-bit | 33% smaller files | Slightly lower dynamic range |
| Shorter render (3s vs 4s) | ~25% smaller | May cut off slow pads |
| FLAC compression in XPN | ~50% smaller | MPC may not support — test first |

**Default strategy:** Every minor 3rd, single velocity, 24-bit, 4s render. This balances quality with ~5MB per preset (~5GB for full library).

## 12. Integration with PresetManager

The XPN export system hooks into the XOmnibus PresetManager:

```cpp
class XPNExporter {
public:
    struct RenderSettings {
        double sampleRate = 44100.0;
        int bitDepth = 24;
        NoteSelection noteSelection = NoteSelection::EveryMinor3rd;
        int velocityLayers = 1;
        float renderLengthSeconds = 4.0f;
        float tailSeconds = 2.0f;
    };

    // Export a bundle
    void exportBundle(const BundleConfig& config,
                      const RenderSettings& settings,
                      const juce::File& outputDir,
                      std::function<void(float progress, juce::String status)> callback);

    // Render a single preset to WAVs
    void renderPreset(const PresetState& preset,
                      const RenderSettings& settings,
                      const juce::File& outputDir);

    // Build XPM for a rendered preset
    void buildXPM(const juce::String& presetName,
                  const juce::File& wavDir,
                  const juce::File& outputXPM,
                  bool isDrumKit = false);

    // Package into XPN
    void packageXPN(const juce::File& sourceDir,
                    const ManifestInfo& manifest,
                    const juce::File& outputXPN);

    // Validate an exported XPN
    ValidationResult validateXPN(const juce::File& xpnFile);

private:
    // CRITICAL: These rules are non-negotiable
    static constexpr bool KEY_TRACK = true;
    static constexpr int ROOT_NOTE = 0;
    static constexpr int EMPTY_VEL_START = 0;

    PresetManager& presetManager;
    MegaToolProcessor& processor;
    juce::File cacheDir;
};
```

**Key constraint:** Rendering runs on a worker thread, NOT the audio thread. The processor is temporarily taken offline for rendering, or a second processor instance is created for offline render.
