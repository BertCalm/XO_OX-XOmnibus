## 1.1 What XPN Actually Is

Pull up any XPN expansion pack in a file manager and rename the extension from `.xpn` to `.zip`. Open it. You will find a directory structure that looks something like this:

```
ONSET_808_Reborn_v1.0/
├── expansion.json
├── Programs/
│   ├── ONSET_808_Reborn_Kick.xpm
│   ├── ONSET_808_Reborn_Snare.xpm
│   └── ONSET_808_Reborn_Full_Kit.xpm
├── Samples/
│   ├── onset_808reborn_kick_v1.wav
│   ├── onset_808reborn_kick_v2.wav
│   ├── onset_808reborn_kick_v3.wav
│   ├── onset_808reborn_kick_v4.wav
│   ├── onset_808reborn_snare_v1.wav
│   └── ... (continuing for all voices and velocity layers)
└── Artwork/
    └── ONSET_808_Reborn_cover.png
```

That's it. The XPN format is a ZIP archive containing three things: a manifest file (`expansion.json`), a folder of XPM program files, and a folder of WAV samples. The artwork folder is optional but you should always include it.

**This is not a streaming format.** The MPC does not connect back to XO_OX's servers. There is no generative component, no live synthesis, no real-time parameter control happening inside the expansion pack itself. Everything is pre-rendered audio. The samples were produced by running XOlokun's synthesizer engines in an offline audio rendering context, capturing the output to WAV files, and packaging those WAV files into this archive.

This distinction matters. When you load an XO_OX expansion pack on your MPC, you are loading audio files that were rendered from the same synthesis engines you could run yourself in the XOlokun plugin. The difference is that in XOlokun, the synthesis happens in real time — every parameter can be adjusted, every coupling relationship is live, every LFO is running. In the XPN pack, all of that has been captured at a single point in time (or, for velocity layers, at several carefully chosen points in time) and frozen as audio.

**The MPC reads it like a library.** When you navigate to an expansion pack in the MPC's browser, the firmware reads `expansion.json` to get the pack metadata and program list, then lets you browse the individual `.xpm` programs. When you load a program, the MPC reads the `.xpm` file to understand the pad-to-sample mapping, velocity layer configuration, envelope settings, and Q-Link assignments. The actual sample audio lives in the WAV files in the `Samples/` directory.

Nothing about this process involves the XOlokun plugin. The XPN pack is completely standalone. You can take an XPN pack and load it on any Akai MPC hardware running firmware 2.x or later without any plugins installed.

**Why this matters for production workflow:** Understanding that XPN packs are frozen audio means understanding their strengths and limits. The strength: you get exactly the sound that was designed, with precisely calibrated velocity layers, envelope settings that were tested on actual hardware, and Q-Link assignments that have been laid out for ergonomic performance. The limit: if you want to reach into the underlying synthesis and change the filter model, you need to go back to XOlokun. The expansion pack gives you the sound; the plugin gives you the source.

For most producers, this is exactly the right tradeoff. You want to make music, not fiddle with synthesis parameters. Load the pack, play the pads, route it into your sequence, done.

## 1.2 The XPM File Anatomy

The heart of an XPN pack is the collection of `.xpm` files. Each `.xpm` file is an XML document that defines one MPC program — either a Drum program or a Keygroup program. (There are also Clip programs and Plugin programs, but we'll get to those later.)

Let me walk through a complete, annotated Drum program XPM. This is based on the actual structure generated by XO_OX's `xpn_drum_export.py` tool, which means it is production-accurate, not a simplified teaching example:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<MPCVObject>
  <Version MajorVersion="2" MinorVersion="1" PatchVersion="0" />
  <Program type="Drum" name="ONSET 808 Reborn — Full Kit">

    <!-- ============================================================
         Program-level settings
    ============================================================ -->
    <ProgramName>ONSET 808 Reborn — Full Kit</ProgramName>
    <Tempo>120.000000</Tempo>
    <PolyPhony>16</PolyPhony>
    <SimultaneousNoteCount>16</SimultaneousNoteCount>

    <!-- ============================================================
         Instruments (128 total; only active pads shown here)
         Each <Instrument> corresponds to one pad.
    ============================================================ -->
    <Instruments>

      <!-- Pad 1: Kick drum (MIDI note 36 / C2) -->
      <Instrument number="0">
        <PadNumber>36</PadNumber>           <!-- GM MIDI note number -->
        <VoiceName>V1 Kick</VoiceName>      <!-- Display name on MPC screen -->
        <Volume>1.0</Volume>                <!-- 0.0–1.0 -->
        <Pan>0.0</Pan>                      <!-- -1.0 (full left) to 1.0 (full right) -->
        <Mute>False</Mute>
        <Solo>False</Solo>
        <MuteGroup>0</MuteGroup>            <!-- 0 = no mute group -->
        <MuteTarget>-1</MuteTarget>
        <OneShot>True</OneShot>             <!-- True = plays to end regardless of note-off -->
        <PolyphonyCount>1</PolyphonyCount>  <!-- Max voices for this pad -->

        <!-- Velocity layers: four layers for kick (pp / mp / mf / ff) -->
        <Layers>

          <!-- Layer 0: pianissimo (velocity 1–39) -->
          <Layer number="0">
            <SampleFile>Samples/onset_808reborn_kick_v1.wav</SampleFile>
            <VelStart>1</VelStart>          <!-- CRITICAL: see Golden Rule #3 -->
            <VelEnd>39</VelEnd>
            <SampleStart>0</SampleStart>    <!-- 0 = file start -->
            <SampleEnd>-1</SampleEnd>       <!-- -1 = file end -->
            <Loop>False</Loop>
            <LoopStart>0</LoopStart>
            <LoopEnd>-1</LoopEnd>
            <Volume>0.85</Volume>           <!-- Layer-level gain scaling -->
            <Pan>0.0</Pan>
            <Tune>0</Tune>                  <!-- Semitone offset -->
            <FineTune>0</FineTune>          <!-- Cent offset -->
            <RootNote>0</RootNote>          <!-- CRITICAL: 0 = MPC auto-detect -->
            <KeyTrack>True</KeyTrack>       <!-- CRITICAL: see Golden Rule #1 -->
            <ZonePlay>1</ZonePlay>          <!-- 1=Velocity, 2=Cycle, 3=Random, 4=RandNoRepeat -->
            <CycleType>0</CycleType>
            <CycleGroup>0</CycleGroup>
          </Layer>

          <!-- Layer 1: mezzo-piano (velocity 40–79) -->
          <Layer number="1">
            <SampleFile>Samples/onset_808reborn_kick_v2.wav</SampleFile>
            <VelStart>40</VelStart>
            <VelEnd>79</VelEnd>
            <!-- ... same structure as Layer 0 ... -->
            <Volume>0.90</Volume>
            <Pan>0.0</Pan>
            <Tune>0</Tune>
            <FineTune>0</FineTune>
            <RootNote>0</RootNote>
            <KeyTrack>True</KeyTrack>
            <ZonePlay>1</ZonePlay>
            <CycleType>0</CycleType>
            <CycleGroup>0</CycleGroup>
          </Layer>

          <!-- Layer 2: mezzo-forte (velocity 80–109) -->
          <Layer number="2">
            <SampleFile>Samples/onset_808reborn_kick_v3.wav</SampleFile>
            <VelStart>80</VelStart>
            <VelEnd>109</VelEnd>
            <Volume>0.95</Volume>
            <Pan>0.0</Pan>
            <Tune>0</Tune>
            <FineTune>0</FineTune>
            <RootNote>0</RootNote>
            <KeyTrack>True</KeyTrack>
            <ZonePlay>1</ZonePlay>
            <CycleType>0</CycleType>
            <CycleGroup>0</CycleGroup>
          </Layer>

          <!-- Layer 3: fortissimo (velocity 110–127) -->
          <Layer number="3">
            <SampleFile>Samples/onset_808reborn_kick_v4.wav</SampleFile>
            <VelStart>110</VelStart>
            <VelEnd>127</VelEnd>
            <Volume>1.0</Volume>
            <Pan>0.0</Pan>
            <Tune>0</Tune>
            <FineTune>0</FineTune>
            <RootNote>0</RootNote>
            <KeyTrack>True</KeyTrack>
            <ZonePlay>1</ZonePlay>
            <CycleType>0</CycleType>
            <CycleGroup>0</CycleGroup>
          </Layer>

        </Layers>

        <!-- Per-pad color (MPC 3.x) — engine accent color -->
        <PadColor>
          <Red>0</Red>
          <Green>102</Green>
          <Blue>255</Blue>
        </PadColor>

      </Instrument>

      <!-- Pad 2: Snare (MIDI note 38 / D2) — abbreviated for space -->
      <Instrument number="1">
        <PadNumber>38</PadNumber>
        <VoiceName>V2 Snare</VoiceName>
        <!-- ... layers for snare v1–v4 ... -->
      </Instrument>

      <!-- ... remaining voices follow the same pattern ... -->

    </Instruments>

    <!-- ============================================================
         Q-Link Assignments (up to 16 in MPC 3.x)
    ============================================================ -->
    <Qlnk>
      <QlnkAssignment number="0">
        <ParameterName>VolumeAttack</ParameterName>
        <Label>ATTACK</Label>              <!-- Max 8 chars for OLED display -->
        <PadNumber>-1</PadNumber>          <!-- -1 = apply to all pads -->
        <Minimum>0.0</Minimum>
        <Maximum>1.0</Maximum>
        <Bipolar>False</Bipolar>           <!-- True = center = 0, False = 0 to max -->
        <Smoothing>20</Smoothing>          <!-- ms of parameter smoothing -->
      </QlnkAssignment>
      <!-- Q-Links 1–3 continuing with DECAY, SUSTAIN, RELEASE -->
    </Qlnk>

    <!-- ============================================================
         PadNoteMap: maps MIDI note numbers to pad positions
    ============================================================ -->
    <PadNoteMap>
      <Entry padNum="0" note="36" />   <!-- Pad A01 = C2 = Kick -->
      <Entry padNum="1" note="38" />   <!-- Pad A02 = D2 = Snare -->
      <Entry padNum="2" note="42" />   <!-- Pad A03 = F#2 = Closed Hat -->
      <Entry padNum="3" note="46" />   <!-- Pad A04 = A#2 = Open Hat -->
      <Entry padNum="4" note="39" />   <!-- Pad B01 = D#2 = Clap -->
      <Entry padNum="5" note="41" />   <!-- Pad B02 = F2 = Tom -->
      <Entry padNum="6" note="43" />   <!-- Pad B03 = G2 = Perc -->
      <Entry padNum="7" note="49" />   <!-- Pad B04 = C#3 = FX -->
    </PadNoteMap>

    <!-- PadGroupMap: groups pads for muting (hat open/closed) -->
    <PadGroupMap>
      <Entry padNum="2" group="1" />   <!-- Closed Hat in mute group 1 -->
      <Entry padNum="3" group="1" />   <!-- Open Hat in mute group 1 -->
    </PadGroupMap>

  </Program>
</MPCVObject>
```

That's a production-ready drum program. Let me call out the key fields you need to understand:

**`<MPCVObject>` and `<Version>`** — The root element of every XPM file. The version numbers correspond to the MPC firmware generation. MPC 3.x files typically use `MajorVersion="2" MinorVersion="1"` in the layer format, even though the firmware version is 3.x. The versioning convention is an internal MPC format version, not a direct firmware version. Do not change these numbers manually.

**`<PadNumber>`** — The MIDI note number this instrument responds to. For drum programs, this determines which physical pad triggers the instrument. The MPC uses the GM MIDI standard convention: 36 = kick (C2), 38 = snare (D2), 42 = closed hat (F#2), 46 = open hat (A#2), 39 = clap (D#2). You can deviate from GM if you have a reason to, but GM convention means your drum programs will respond correctly to external MIDI devices and MPC's own GM drum pad layout.

**`<VoiceName>`** — What appears on the MPC's screen when the pad is selected. Keep this short and descriptive. This is different from `<ProgramName>`, which is the name of the entire program.

**`<OneShot>`** — Set to `True` for percussion. When `OneShot=True`, the MPC plays the sample to completion regardless of note-off messages. When `False`, note-off stops playback. You almost always want `True` for drum hits. For sustained melodic content in keygroup programs, you want `False`.

**`<Layers>` and `<Layer>`** — The core of the velocity system. Each layer defines one sample and the velocity range it plays in. The `VelStart` and `VelEnd` fields define the range (1–127). Layers are selected from highest `VelEnd` downward — the MPC finds the first layer whose range contains the incoming velocity. More on this architecture in section 1.4.

**`<SampleFile>`** — The relative path to the WAV file within the XPN archive. The path is relative to the root of the ZIP archive, not to the `.xpm` file's location. Get this wrong and the MPC shows a missing sample error.

**`<ZonePlay>`** — How multiple layers in a velocity zone are handled. `1` = velocity selection (standard — pick the layer matching the incoming velocity). `2` = cycle (round-robin — alternate through layers on successive hits). `3` = random (random selection each hit). `4` = random-no-repeat (random, never same twice in a row). This field determines the basic playback behavior independent of velocity layers. A hat pad with four cycle samples would have `ZonePlay=2` on each layer, while a kick pad with four velocity layers would have `ZonePlay=1`.

**`<CycleType>` and `<CycleGroup>`** — These work together with `ZonePlay` to implement round-robin and grouping behavior. `CycleGroup` gives a group identifier so multiple pads can share the same cycle counter (useful when you want related instruments to cycle together).

**`<PadNoteMap>`** — Maps MIDI note numbers to physical pad positions. This is how the MPC knows that when it receives MIDI note 36, it should light up pad A01. Without a `PadNoteMap`, the pads still function but the MIDI note assignments may not match your expectation.

**`<PadGroupMap>`** — Assigns pads to mute groups. Classic use case: closed hat and open hat in the same mute group means hitting the closed hat silences the open hat, exactly like a real hi-hat pedal.

## 1.3 The Three Golden Rules

In the hundreds of test loads and hundreds of hours of pack development at XO_OX, three rules emerged that must never be violated. Breaking any one of them produces a specific, repeatable, maddening failure mode.

I discovered all three the hard way.

---

### Golden Rule #1: `KeyTrack = "True"`

**The rule:** Every layer in a keygroup program must have `<KeyTrack>True</KeyTrack>`.

**What happens when you break it:** The sample plays back at its recorded pitch regardless of which key you press. Hit C3 — correct pitch. Hit E3 — still the same C3 pitch. Hit C5 — still C3. Your entire keygroup program plays the same note across the full range, which is both useless and bewildering if you haven't encountered it before.

**Why this happens:** `KeyTrack=False` is a legitimate feature. It means "ignore the incoming MIDI note and always play this sample at its root pitch." It exists for use cases like drum programs where you want a specific sample file to always play at its rendered pitch, regardless of what MIDI note triggered it. The problem is that in early versions of the Oxport keygroup exporter, this field defaulted to `False` because the drum exporter — which runs first and was built first — legitimately needed `False` for some percussion voices. The keygroup exporter was built by copying some boilerplate from the drum exporter.

I launched a finished keygroup program of OVERDUB's warm string patches on hardware. Played a chord. Got a single-note cluster of the exact same pitch repeated six times. Stared at it for twenty minutes before realizing the issue.

**The fix in Oxport:** `xpn_keygroup_export.py` now sets `KeyTrack=True` for every layer unconditionally. The drum exporter sets `KeyTrack=False` or `KeyTrack=True` per voice based on `VOICE_DEFAULTS["key_track"]`. There is a QA check in `xpn_validator.py` that flags keygroup programs with any `KeyTrack=False` layer.

---

### Golden Rule #2: `RootNote = "0"`

**The rule:** Set `<RootNote>0</RootNote>` in every layer. Do not hardcode a specific MIDI note number as the root.

**What `RootNote=0` means:** It is not "the note is C-1 (MIDI note 0)." It is the MPC convention for "auto-detect the root note." The MPC firmware will look at the sample filename for a pitch indicator (e.g., `odyssey_drift_C3.wav` contains "C3"), detect the pitch programmatically, or fall back to a sensible default. `RootNote=0` is the signal to the firmware: "I am trusting you to figure this out."

**What happens when you break it:** If you hardcode `RootNote=60` (C3) for every sample in a keygroup, the MPC will pitch-shift every sample as if it needs to be transposed from C3 to the target key. A sample of a note rendered at A3 will be pitch-shifted down a minor third to arrive at the target key, rather than being recognized as an A3 sample and pitch-shifted correctly from A3. The entire keygroup plays out of tune relative to itself — low notes are flat, high notes are sharp, and the chromatic relationship is completely wrong.

**Why people hardcode it:** Because it feels more explicit and "correct" to state the root note. It seems like the responsible thing to do. In most sampler formats (Kontakt, EXS24, SFZ), explicitly setting the root note is best practice. In the MPC's XPN format, `RootNote=0` with correctly named sample files is better practice because it lets the MPC's pitch detection resolve any ambiguity.

**The naming convention that makes this work:** XO_OX sample files use the format `{preset_slug}_{voice}_{note}_{variant}.wav` — for example, `odyssey_drift_C3_v2.wav`. The `_C3_` component gives the MPC's filename parser what it needs to identify the root note. The `xpn_render_spec.py` tool enforces this naming convention for all rendered samples.

**The one exception:** Drum samples that are intentionally not pitched — noise-dominant kicks, claps, hi-hats — should use a nominal root note of C2 (MIDI 48) in the filename for organizational consistency, even though the MPC will detect them correctly regardless.

---

### Golden Rule #3: `VelStart = "0"` for Unused Layers

**The rule:** Any declared layer that has no sample assigned must have `<VelStart>0</VelStart>`. Do not leave empty layers with non-zero `VelStart` values.

**What happens when you break it:** Ghost triggering. You hit a pad at MIDI velocity 45. The MPC checks its layer list. Layer 0 has `VelStart=1`, `VelEnd=39`, sample assigned. Layer 1 has `VelStart=40`, `VelEnd=79`, but no sample assigned — `<SampleFile></SampleFile>`. The MPC still selects layer 1 based on the velocity range. With no sample to play, it plays... something. Sometimes silence. Sometimes the last loaded sample in that slot from a previous program load. On certain firmware versions, it plays whatever audio happens to be sitting in the buffer.

**Why this happens:** When building an XPM programmatically, it is tempting to generate a full complement of layers with velocity ranges defined, then populate sample paths for only the layers that have actual audio. The XPM schema technically allows empty `<SampleFile>` tags. But the MPC firmware does not handle this gracefully — it selects the layer based on velocity range before checking whether a sample is actually assigned.

**The fix:** In `xpn_drum_export.py` and `xpn_keygroup_export.py`, the Oxport tools generate only as many layers as there are actual WAV files. A 1-sample pad gets 1 layer with `VelStart=1, VelEnd=127`. A 4-sample velocity kit gets 4 layers with the appropriate splits. No empty layers, no `VelStart` mismatch.

---

> **Sidebox — The Three Golden Rules at a Glance**
>
> `KeyTrack = "True"` — samples must transpose, or a keygroup is useless
>
> `RootNote = "0"` — let MPC auto-detect, or pitch relationships break
>
> `VelStart = "0"` for empty layers — or ghost triggering occurs
>
> These are not preferences. They are correctness requirements.

## 1.4 Velocity Layer Architecture

Velocity layers are where XPN packs become musically alive or musically dead. Get them right and a producer feels like they have an instrument. Get them wrong and they feel like they have a sample player.

The doctrinal principle at XO_OX — Doctrine D001 — states: **"Velocity must shape timbre, not just amplitude."** If the only difference between a soft hit and a hard hit is the volume, you have not designed velocity layers. You have designed a loudness knob.

What should change with velocity:

- **Filter cutoff** — the clearest timbral velocity response. Soft hits: filter closed, fundamental dominant, dull and intimate. Hard hits: filter wide open, harmonics present, bright and aggressive.
- **Noise content** — in percussion engines, velocity should add noise burst, attack click, and mechanical transient energy proportional to velocity. A quiet hit is clean; a loud hit is complex.
- **Saturation** — in bass and drive engines, high velocity crosses the saturation threshold. The sample changes character, not just level.
- **Attack speed** — harder hits have faster perceived attacks. Some engines (OBLONG, OBESE) render noticeably different attack envelopes at high velocity.

**The 1-layer approach** is correct for one type of content: pure texture and atmospheric pads where velocity is irrelevant to the user's workflow. If a producer is going to set a pad volume once and leave it, a single high-quality sample is better than four layers rendered at "wrong" parameter states.

**The 4-layer standard** is the minimum for any instrument you want players to express through. Splits at approximately 1-39 / 40-79 / 80-109 / 110-127. Render at MIDI velocities 20, 50, 80, 120. The exact split points matter less than rendering at representative velocities within each zone.

**The 8-layer pro design** is for flagship instruments and instruments with dramatic velocity response curves. ONSET drum voices, OBESE bass programs, ORACLE pitched percussion — any instrument where the filter sweep or saturation curve is strongly velocity-dependent benefits from 8 layers because you capture the curve more accurately.

For 8-layer designs, split points: 1-19, 20-39, 40-59, 60-79, 80-99, 100-109, 110-119, 120-127. Render at: 10, 30, 50, 70, 90, 105, 115, 124.

**The timbral shift approach** is a design philosophy, not a different technical implementation. It means that when you choose parameter values for each velocity layer render, the goal is timbral change, not proportional level change. The XPN tool will handle level normalization across layers. Your job as the sound designer is to ensure that the low-velocity and high-velocity renders sound like meaningfully different versions of the same instrument — different in character, not just volume.

### `ZonePlay` Values

`ZonePlay=1` (Velocity) — the standard for melodic programs and most drum voices. The MPC selects the layer based on incoming velocity. Higher velocity = higher-index layer (assuming layers are ordered by VelStart ascending).

`ZonePlay=2` (Cycle) — round-robin. The MPC cycles through all declared layers on successive hits, regardless of velocity. Essential for hi-hat programs and any voice where you want variation to prevent the "machine gun" effect (identical sample triggering at high BPMs sounds robotic).

`ZonePlay=3` (Random) — random selection. Each hit picks a random layer. Good for clap and organic percussion voices where occasional variation feels natural but cycle predictability is undesirable.

`ZonePlay=4` (Random No Repeat) — random, never same twice in a row. Slightly more sophisticated than pure random; prevents the one-in-N chance that you hear the same sample twice consecutively.

### `CycleType` and `CycleGroup`

For round-robin behavior, `CycleType` sets the mode (0 = velocity-based sequential, 1 = round-robin cycle, 2 = random, 3 = random-no-repeat) and `CycleGroup` assigns the layer to a numbered group. Layers with the same `CycleGroup` value share a cycle counter — they advance the round-robin together. This is the mechanism that allows XPN drum kits to cycle hi-hat variations in sync.

### Sonic DNA Velocity Curves

XO_OX's 6D Sonic DNA system includes an `aggression` axis (0.0–1.0) that directly predicts the appropriate velocity layer architecture:

| Aggression | Layers | Curve | Filter Env Amount |
|------------|--------|-------|------------------|
| 0.0–0.2 | 4 | Logarithmic (gentle) | Low (filter barely moves) |
| 0.3–0.5 | 4–6 | Linear | Medium |
| 0.6–0.7 | 6–8 | Linear | Medium-High |
| 0.8–1.0 | 8 | Exponential (steep) | High |

High-aggression engines (OBLONG, OBESE, ONSET at peak drive, OUROBOROS at high chaos) have dramatic velocity response curves. The difference between velocity 1 and velocity 127 should be a completely different character, not a slightly louder version of the same thing.

## 1.5 Program Types

The XPN format supports multiple program types. Each type tells the MPC firmware how to interpret and play back the program. Choosing the wrong type for your content is a fundamental design error.

### Drum Programs

`<Program type="Drum">` — 128 instrument slots, each triggered by a specific MIDI note number. This is the right type for:

- Percussion kits (kicks, snares, hats, claps)
- Non-pitched one-shot samples
- Rhythm instruments that do not need key-tracked pitch transposition

The defining feature of drum programs is that each instrument slot has its own complete sample stack, envelope, and routing. Slot 36 (kick) is completely independent from slot 38 (snare). Each can have its own velocity layers, mute groups, and Q-Link assignments.

Drum programs use `PadNoteMap` to map MIDI notes to physical pad positions, and `PadGroupMap` for mute group relationships.

### Keygroup Programs

`<Program type="Keygroup">` — pitched instruments designed for melodic playback across the keyboard. This is the right type for:

- Melodic synthesizer patches (OVERDUB, ODYSSEY, ORBITAL, OPAL)
- Bass instruments
- Chord pads and atmospheric textures
- Any content where you want pitch transposition as you play across the keyboard

In a keygroup program, each `<Keygroup>` element defines a zone of the keyboard (defined by `<LowKey>` and `<HighKey>`) with its own sample, root note, and envelope. The MPC pitch-shifts each sample to cover the zone from its root note to the zone boundaries.

The standard XO_OX strategy renders samples at every minor third across the intended range (e.g., C1, Eb1, F#1, A1, C2, Eb2...) — 21 sample points covering C1 to C6. Each sample covers approximately ±1.5 semitones of pitch-shifting in either direction. This keeps the stretch ratio well within the quality threshold for synthesis-rendered audio.

Keygroup programs are where `KeyTrack=True` is non-negotiable (see Golden Rule #1) and where loop points become important for sustained patches.

### Clip Programs

`<Program type="Clip">` — looped audio content, typically full-bar or half-bar samples intended to be played back with MPC's time-stretching engine. This is the right type for:

- Loop-based content
- Stem exports
- Rhythmic audio content that should stretch to the project tempo

XO_OX does not currently generate Clip programs from XOlokun renders. The Oxport pipeline is focused on Drum and Keygroup formats. Clip programs are a potential V2 addition for loop-based collection releases.

**The distinction matters for pack design decisions.** A OVERDUB patch that you want producers to play melodically → Keygroup program. The same OVERDUB engine rendered as a one-bar evolving loop → Clip program. A cymbal pad with no pitch intention → Drum program. Getting the type wrong means the producer's workflow breaks: they cannot pitch a drum program instrument, cannot use a keygroup program for one-shots efficiently, and cannot time-stretch a drum program.

## 1.6 The XPN Manifest (expansion.json)

Every XPN archive contains a file named `expansion.json` at the root level. This is the manifest — the document that tells the MPC what is in the pack and how to display it in the browser.

A minimal valid manifest:

```json
{
  "schemaVersion": "3.0",
  "name": "ONSET 808 Reborn",
  "vendor": "XO_OX Designs",
  "version": "1.0.0",
  "description": "8-voice percussion synthesis from XOlokun ONSET engine. Electric Blue drums designed for modern hip-hop and electronic production.",
  "tags": ["drums", "percussion", "808", "electronic", "hip-hop"],
  "targetHardware": ["MPC Live II", "MPC One", "MPC Key 61", "MPC Live III"],
  "sampleRate": 44100,
  "programs": [
    {
      "name": "ONSET 808 Reborn — Full Kit",
      "file": "Programs/ONSET_808_Reborn_Full_Kit.xpm",
      "type": "Drum"
    },
    {
      "name": "ONSET 808 Reborn — Kick Only",
      "file": "Programs/ONSET_808_Reborn_Kick.xpm",
      "type": "Drum"
    }
  ],
  "artwork": "Artwork/ONSET_808_Reborn_cover.png",
  "previewAudio": "Artwork/ONSET_808_Reborn_preview.wav"
}
```

**Required fields:**

- `name` — Pack name. Appears in the MPC browser. Keep it under 40 characters.
- `vendor` — Creator or company name.
- `version` — Semantic version string. Use `MAJOR.MINOR.PATCH` format. `1.0.0` for initial release, `1.0.1` for bug fixes, `1.1.0` for content additions.
- `description` — Free-text description. The MPC displays this in the pack info panel.
- `programs` — Array of program entries. Each entry must have `name`, `file` (relative path), and `type`.

**Recommended fields:**

- `schemaVersion` — The XPN schema version the manifest targets. Use `"3.0"` for all current production.
- `tags` — Array of genre/content tags used for filtering in the browser.
- `targetHardware` — Array of hardware models the pack was tested on.
- `artwork` — Path to 1000×1000 PNG cover image.
- `previewAudio` — Path to a short preview WAV (MPC 3.x feature; plays when the pack is selected in browser without loading).
- `sampleRate` — Sample rate of the included WAV files (44100 or 48000).

### The `_xo_ox` Extension Namespace

XO_OX uses a custom extension namespace within `expansion.json` to carry metadata that the standard manifest fields do not support:

```json
{
  "schemaVersion": "3.0",
  "name": "ONSET 808 Reborn",
  ...
  "_xo_ox": {
    "engine": "ONSET",
    "engineAccentColor": "#0066FF",
    "sonicDNA": {
      "brightness": 0.7,
      "warmth": 0.4,
      "movement": 0.5,
      "density": 0.6,
      "space": 0.4,
      "aggression": 0.8
    },
    "couplingNote": "XVC cross-voice coupling baked into full-kit renders. Individual voice programs are decoupled for stem workflow.",
    "fieldGuideUrl": "https://xo-ox.org/guide/onset-808-reborn",
    "ebookSection": "Chapter 2.6"
  }
}
```

MPC firmware ignores any fields it does not recognize, so the `_xo_ox` block is silently passed over on hardware. But it is there for XO_OX tooling (the website, the Oxport pipeline, future XOlokun integration) to read.

### Version Numbering

Semantic versioning (`MAJOR.MINOR.PATCH`) is the right choice for expansion packs:

- **PATCH** increment: bug fixes, sample replacement without content change, metadata corrections
- **MINOR** increment: new programs added to the pack, new velocity layers added to existing programs, new Q-Link assignments
- **MAJOR** increment: complete redesign, breaking change to program structure, rename of programs

On MPC hardware, if a producer has `ONSET 808 Reborn v1.0.0` loaded in a project, loading `v1.1.0` into the same slot should work without changes to their project. Incrementing `MAJOR` version may break project recall if program names have changed.

## 1.7 MPC 3.x Updates Affecting XPN

The MPC firmware has evolved significantly from the 2.x era to the current 3.x series. Understanding which features are new and which are baseline helps you decide what to use now versus what to plan for.

### Pad Color (RGB Assignment)

MPC hardware has had RGB pad coloring for years, but MPC 3.x formalized the serialization of pad color in XPM files. Two XML encoding styles appear in community-shared XPMs:

```xml
<!-- Child element form (more verbose, unambiguous) -->
<PadColor>
  <Red>0</Red>
  <Green>102</Green>
  <Blue>255</Blue>
</PadColor>

<!-- Attribute form (more compact) -->
<PadColor red="0" green="102" blue="255" />
```

The attribute form appears to be the MPC 3.x standardized encoding. XO_OX uses the child element form for readability in generated files; both forms load correctly.

For XO_OX packs, the convention is: pad colors match the engine accent color. ONSET packs use Electric Blue `#0066FF` (R=0, G=102, B=255). OPAL packs use Lavender `#A78BFA` (R=167, G=139, B=250). This gives instant visual identity on the pad grid — you know which engine you are playing before you hit a pad.

### BendRange and Pitch Wheel

MPC 3.x programs support explicit pitch bend range configuration at the program level. The field `<BendRange>` (in semitones) can be set in keygroup programs to override the global MIDI pitch wheel range. For most XO_OX melodic programs, `<BendRange>2</BendRange>` (±2 semitones) is the standard. Lead programs and bass programs may use `<BendRange>12</BendRange>` for expressive pitch slides.

### Insert Effects (4-Slot FX Chain)

MPC 3.x added a 4-slot insert FX chain per program, serialized in XPM as:

```xml
<InsertFX>
  <FXSlot number="0">
    <PluginName>AIR Tube Drive</PluginName>
    <Enable>False</Enable>         <!-- Disabled by default; user enables -->
    <Parameters>
      <Drive>0.3</Drive>
      <Tone>0.5</Tone>
    </Parameters>
  </FXSlot>
  <!-- ... slots 1–3 ... -->
</InsertFX>
```

XO_OX currently ships programs with FX slots disabled by default (useful as "turn on and go" options) rather than pre-enabled effects that affect the sample playback. The philosophy: the samples were already rendered with appropriate processing from the engine. Adding more FX at the program level is the producer's choice. We suggest rather than impose.

### MPCe Quad-Corner Architecture

The MPCe hardware (2026) introduces pressure-sensitive pad corners. Each of the four corners of a pad can carry a separate assignment — this maps conceptually to a 4-trigger-point-per-pad system. In XPN terms, the anticipatory field is `<PadCornerAssignment>` per instrument. The schema for this is not yet finalized in the public documentation.

XO_OX's `xpn_mpce_quad_builder.py` tool generates these assignments for MPCe-optimized packs: the four corners of a kick pad might assign velocity zone, tuning variation, and reverb send to the four touch points. This is forward-looking design — it prepares packs for full MPCe capabilities as the firmware matures. Current MPC Live II/III hardware ignores these fields silently.

The practical implication for pack design now: engineer your sample architecture so that the four most useful performance variations of each instrument map naturally to four pad corners. Think: low corner = soft/dry, right corner = bright/open, top corner = sustained, left corner = reverb send. Design the pads so these variations are sonically distinct and musically useful before the corner architecture is even active.

## 1.8 The Oxport Pipeline

Knowing the XPN format is one thing. Producing a complete, tested, hardware-verified XPN pack from scratch is another. XO_OX developed the Oxport pipeline — a collection of Python tools in the `/Tools/` directory of the XOlokun repository — to automate the mechanical parts of pack production while keeping human judgment in the loop for the parts that require it.

The tools are designed to be run individually or orchestrated by the `oxport.py` master script.

### The Tools

**`xpn_render_spec.py`** — The front door. Given an XOlokun engine name and preset, generates a JSON render specification: what notes to render, at what velocities, what sample rate, how long each sample should be, whether to render mono or stereo, whether loops are needed, and what the expected DNA profile is. This spec drives all subsequent stages.

```bash
python3 Tools/xpn_render_spec.py \
  --engine Onset \
  --preset "808 Reborn" \
  --output render_specs/onset_808reborn.json
```

**`xpn_sample_categorizer.py`** — Takes a folder of raw WAV files (your rendered output from XOlokun or any offline renderer) and sorts them into the expected directory structure with correct naming conventions. Reads the render spec to know what to expect. Flags any missing or incorrectly named files.

```bash
python3 Tools/xpn_sample_categorizer.py \
  --spec render_specs/onset_808reborn.json \
  --wavs-dir ~/renders/onset_808reborn_raw/ \
  --output categorized/onset_808reborn/
```

**`xpn_drum_export.py`** — The drum program generator. Takes a categorized sample directory and a kit configuration, generates one or more `.xpm` Drum program files. Handles velocity layer architecture, round-robin assignment, mute group configuration, Q-Link assignments, and pad color embedding.

```bash
python3 Tools/xpn_drum_export.py \
  --preset "808 Reborn" \
  --mode smart \
  --wavs-dir categorized/onset_808reborn/ \
  --output-dir programs/onset_808reborn/
```

The `--mode smart` flag selects the per-voice-optimal playback mode: kick and snare use velocity layers (dynamics matter), hi-hats use cycle (machine-gun prevention), claps use random (organic feel), FX voice uses random-no-repeat (never plays the same FX twice in a row).

**`xpn_keygroup_export.py`** — The melodic program generator. Takes multi-sample root note folders and a configuration JSON, generates `.xpm` Keygroup program files. Handles auto-zone calculation from filename root notes, layer stack generation, loop point configuration, LFO stub generation, and 4-slot FX chain stubs.

```bash
python3 Tools/xpn_keygroup_export.py \
  --preset "Analog Warmth" \
  --engine Overdub \
  --wavs-dir categorized/overdub_analog_warmth/ \
  --output-dir programs/overdub_analog_warmth/
```

**`xpn_kit_expander.py`** — Takes a seed kit and generates N variants by applying parameter variations: pitch offset, decay time, velocity curve shape, and layer shuffle. Used when you want to release a collection of related kits (e.g., eight flavors of a snare kit) without rendering each one from scratch.

**`xpn_qa_checker.py`** — Perceptual QA on the rendered WAV files before they go into programs. Checks: sample peak levels within acceptable range, velocity layer transitions are smooth (no audible jump), loop points are clean (for looped content), silence trimming is correct, and no unexpected DC offset. Outputs a pass/fail report with file-level detail.

```bash
python3 Tools/xpn_qa_checker.py \
  --wavs-dir categorized/onset_808reborn/ \
  --spec render_specs/onset_808reborn.json \
  --report qa_reports/onset_808reborn.txt
```

**`xpn_cover_art.py`** — Generates branded procedural cover art using the engine accent color, pack name, and XO_OX design system. Outputs two files: a 1200×1200 master PNG and a 1000×1000 XPN-embed version. The design system uses warm white `#F8F6F3` as background, the engine accent color as primary, and Space Grotesk typography.

```bash
python3 Tools/xpn_cover_art.py \
  --engine Onset \
  --pack-name "808 Reborn" \
  --accent-color "#0066FF" \
  --output artwork/onset_808reborn/
```

**`xpn_bundle_builder.py`** — Assembles the final XPN ZIP archive. Takes a folder of `.xpm` files, a `Samples/` directory, a manifest JSON, and an `Artwork/` folder. Performs a final integrity check (all referenced samples exist, manifest schema validates) before zipping. Outputs the `.xpn` file.

```bash
python3 Tools/xpn_bundle_builder.py \
  --programs programs/onset_808reborn/ \
  --samples categorized/onset_808reborn/ \
  --manifest manifests/onset_808reborn.json \
  --artwork artwork/onset_808reborn/ \
  --output releases/ONSET_808_Reborn_v1.0.xpn
```

**`xpn_packager.py`** — Wraps the bundle in a release package: adds a README, changelog, install instructions, and generates a checksum file for integrity verification. This is the file that goes to Patreon or the XO-OX.org download page.

**`xpn_validator.py`** — Validates a manifest JSON or XPM file against the schema. Checks required fields, field types, value ranges, and XO_OX-specific conventions (Q-Link names ≤8 characters, pad colors are engine-appropriate, etc.). Intended for use in CI pipelines and as a pre-bundle check.

### Running the Full Pipeline

The `oxport.py` orchestrator chains all stages into a single command:

```bash
python3 Tools/oxport.py run \
  --engine Onset \
  --preset "808 Reborn" \
  --wavs-dir ~/renders/onset_808reborn/ \
  --output-dir ~/releases/onset/808_reborn/
```

This runs: `render_spec` → `categorize` → `qa` → `export` → `cover_art` → `package`.

The pipeline has a `--dry-run` flag that prints each stage's planned actions without executing them — useful for confirming the configuration before committing to a full render session.

You can skip individual stages if they have already been completed:

```bash
python3 Tools/oxport.py run \
  --engine Onset \
  --preset "808 Reborn" \
  --wavs-dir ~/renders/ \
  --output-dir ~/releases/ \
  --skip cover_art,categorize
```

**Where human judgment is required:** The pipeline does not make creative decisions for you. The render spec generator produces a specification that reflects known best practices for the engine type, but you must review it before rendering. The QA checker flags technical issues but cannot assess whether the velocity layer transitions feel musically right. Cover art generation produces on-brand output but cannot replace a designer's eye for a hero release.

The rule of thumb for pack production time:
- Drum kit (single engine, 8 voices, 4 layers): approximately 2–3 hours including rendering time
- Melodic keygroup (single preset, minor-3rd multi-sample, 3 velocity layers): approximately 3–4 hours
- Full collection pack (multiple programs, unified art direction, hardware validation): one full day

Hardware validation cannot be skipped. Load the XPN on actual MPC hardware before releasing it. The desktop MPC Beats application is useful for schematic checks but does not replicate every nuance of hardware sample loading, physical pad response, and Q-Link behavior.

Before you get there, though, memorize the three rules. Post them above your monitor. Set them as your phone wallpaper if necessary. The format will forgive a lot of imprecision. It will not forgive `KeyTrack=False` in a keygroup, a non-zero `RootNote` hardcode, or a ghost-trigger-inducing empty layer. Those bugs are invisible until playback, which means they will show up in front of an audience.

Golden Rule #1: `KeyTrack = True`. Golden Rule #2: `RootNote = 0`. Golden Rule #3: `VelStart = 0` for empty layers.

Know them. Live them.
