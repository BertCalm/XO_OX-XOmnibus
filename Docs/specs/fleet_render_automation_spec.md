# Fleet Render Automation — P0.1 Spec

> Status: SPEC READY | Priority: P0.1 (highest pipeline bottleneck)
> Owner: Kai (Rex + Atlas) | Effort: L (1-2 weeks)

## The Problem

Every XPN pack requires rendered WAV samples. Currently 100% manual: open XOlokun,
load preset, play notes, record, rename, repeat. At 34 engines x 150+ presets x 4
velocity layers x multiple notes = hundreds of thousands of WAVs needed.

## Architecture: Hybrid Approach

**Phase 1 (ship now): Python + MIDI + audio loopback**

    oxport render --engine Odyssey --output-dir ./wavs/odyssey/

- XOlokun runs with UI
- Python sends MIDI for note triggering via mido
- Audio captured via loopback (BlackHole macOS, JACK Linux)
- WAV naming follows Oxport conventions automatically
- Real-time only (slow but functional)

**Phase 2 (month 2): JUCE headless render**

    xolokun --headless --render-spec spec.json --output-dir ./wavs/

- Bit-exact, faster than real-time, no audio interface needed
- Reads render spec JSON directly from xpn_render_spec.py
- Deterministic output

## Render Spec Integration

Already generated by xpn_render_spec.py. Fleet Render reads this directly.

## Output Convention

    {output_dir}/{engine_slug}/{preset_slug}__{NOTE}__v{n}.WAV

Matches xpn_keygroup_export.py expectations.

## CLI Integration

    oxport render --engine Odyssey --output-dir ./wavs/    # single engine
    oxport render --spec custom.json --output-dir ./wavs/  # custom spec
    oxport render --fleet --output-dir ./wavs/             # all engines

Full pipeline: oxport render -> oxport run -> oxport validate
