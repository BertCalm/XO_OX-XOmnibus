How Engines Relate

Abstract

The XO_OX Coupling Protocol defines a lightweight standard for inter-engine modulation — a way for one synthesis engine to influence another’s behavior in real time, at control rate, without replacing MIDI or audio routing.

Where MIDI describes what to play and audio busses carry what was played, coupling carries the relationship. An ongoing, continuously-updated signal that says: while this engine is doing that, this other engine should respond like this.

“MIDI says ‘play this note.’ Coupling says ‘relate to this voice like this.’”

Version 0.1 of this specification defines three public coupling types, a binary message format for transmitting coupling data between engines, and a three-tier compliance model for implementors. The reference implementation is XOceanus — a free open-source multi-engine synthesizer by XO_OX Designs implementing all three public types plus twelve extended types (15 total).

Motivation

The gap between MIDI and music

MIDI is good at discrete events: a note starts at this pitch, with this velocity, and stops here. DAW automation extends this to parameter curves. Neither describes the relationship between simultaneously-playing voices.

When a jazz guitarist comps behind a horn player, something more subtle is happening than MIDI note events. The guitarist listens to the horn’s dynamics and pulls back. The horn player hears the chord voicing and adjusts inflection. They are coupled — each voice actively shaping how the other behaves.

Layering two synthesizers on the same MIDI bus gives you additive sound. Coupling them gives you a conversation.

Why this is not modulation

Conventional modulation (LFOs, envelopes, mod matrix slots) is self-contained. Engine A modulates its own parameters according to its own internal state. Coupling is different: the source of modulation is another engine’s output or internal state, observed externally and applied across a relationship boundary.

The distinction matters architecturally. Modulation is monologue. Coupling is dialogue.

Terminology

These terms have precise meanings throughout this specification.

Term Definition
Source Engine The engine whose output or internal state drives a coupling route. Also called the speaker.
Target Engine The engine whose parameters are modulated by coupling input. Also called the listener.
Coupling Type An enum value identifying the semantic relationship. Tells the target how to interpret the incoming signal.
Amount A float in [0.0, 1.0] encoding the depth of the coupling relationship. Scales linearly.
Route A complete relationship: one source, one target, one type, one amount. Multiple routes between a pair are permitted.
Control Rate Coupling updates are delivered at block boundaries (32–512 samples). Not per-sample unless explicitly specified by type.

Core Types — Version 0.1

Version 0.1 publishes three coupling types, assigned permanent type IDs in the 0x01–0x03 range. Implementations claiming v0.1 compliance must support at least one of these types at the appropriate compliance level.

type = 0x01

Sympathetic Coupling

Two engines reinforce each other’s harmonic content. The source engine’s pitch and filter characteristics are mirrored (or harmonically related) to the target, creating parallel resonance. Neither engine dominates — both bloom together.

Like two wine glasses vibrating in sympathetic resonance when one is struck.

type = 0x02

Complementary Coupling

One engine recedes where the other is active. Not mixing-level ducking — timbral and rhythmic complementarity. When the source is loud, the target pulls back or shifts to fill the gap.

Call and response. The rhythm guitarist drops out for the solo because the two players are listening to each other.

type = 0x03

Responsive Coupling

One engine’s dynamics shape the other’s timbre. The source’s envelope trajectory — attack transient, sustain decay, release tail — modulates the target’s filter, wavetable position, or morph parameter.

A pianist’s touch on one note affecting the resonance character of adjacent strings through sympathetic mechanical influence.

0x01 — Sympathetic Coupling

The target engine does not copy the source pitch verbatim. It receives the source pitch as a bias — in semitones relative to its own current pitch — and applies it additively. This preserves the target’s independent voice while pulling it into harmonic alignment.

Musical uses: Pad + lead where the pad brightens as the lead ascends. Two harmonic voices that need to track each other without hard sync. Drone engines that bloom when another voice enters their overtone series.

0x02 — Complementary Coupling

The target receives the source’s amplitude as a choke signal. At amount=1.0, when the source is at full amplitude, the target attenuates to its declared floor. When the source is silent, the target returns to full presence. A floor_level parameter prevents complete disappearance — the target remains audible even when fully choked.

Source Engine └── Amplitude envelope (RMS) ────────────────► Target amplitude attenuation Target blend inversion ↑ amount [0,1] 0 = no choke; 1.0 = full inverse

Musical uses: Percussion + sustained pad where the pad opens in the spaces between hits. Two drums that create a groove by occupying opposite moments. Rhythm engine + melody engine with organic push/pull dynamic.

0x03 — Responsive Coupling

The source engine provides a smoothed envelope following its own amplitude. The target maps this envelope to a declared timbral parameter. The specific parameter mapped is the target’s choice — the source only provides the envelope signal and the scaling amount. Envelope following is performed by the host matrix; the source engine does not need to implement it directly.

Source Engine └── Envelope follower output ────────────────► Target filter cutoff modulation └── (optional) LFO phase ────────────────► Target morph / wavetable position ↑ amount [0,1] scales modulation depth

Musical uses: Drums driving a pad’s filter without the compression artifact. Lead melody shaping a textural layer’s wavetable position. A rhythmic engine whose attack shape animates a sustained engine’s timbre in real time.

Message Format

Coupling data is transmitted as a control-rate update packet — a 16-byte binary structure delivered to target engines at block boundaries. The format is minimal, host-independent, and extensible.

struct CouplingPacket { uint8_t version; // Protocol version. 0x01 for this spec. uint8_t type; // CouplingType (0x01, 0x02, 0x03 for v0.1) uint16_t source_id; // Opaque source engine identifier (host-assigned) uint16_t target_id; // Opaque target engine identifier (host-assigned) float amount; // Coupling depth [0.0, 1.0] float signal; // Current coupling signal value [0.0, 1.0] uint8_t flags; // Reserved. Set to 0x00. uint8_t reserved; // Alignment padding. Set to 0x00. }; // Total: 16 bytes per packet

Field semantics

signal is the live value — updated every block from the source engine. For Sympathetic, it is the source amplitude RMS. For Complementary, it is the source amplitude RMS. For Responsive, it is the source envelope follower output. Always [0.0, 1.0].

amount is the configured depth of the relationship — set at route definition time, typically a user-controlled knob. The target multiplies amount × signal to determine the actual modulation applied.

A target receiving a packet with an unknown type must silently ignore it. A packet with amount = 0.0 is a valid no-op and may be used to signal route deactivation without removing the route.

Compliance Levels

Implementations may declare compliance at one of three levels. Higher levels include all capabilities of lower levels.

Level 1

Listener

Can receive coupling input for at least one public type and apply meaningful modulation in response. Minimum requirement to claim v0.1 compatibility.

Level 2

Speaker

Can emit coupling data for at least one public type. Provides a stable output signal (RMS, envelope, or pitch) normalized to [0.0, 1.0] for the host matrix to route.

Level 3

Full v0.1

Both listener and speaker. All three public types implemented for both input and output. The highest compliance tier for this specification.

Declaration syntax

// Declare in engine metadata, README, or plugin descriptor: COUPLING_COMPLIANCE: Listener SUPPORTED_TYPES: 0x01, 0x02 // list which types you listen to COUPLING_COMPLIANCE: Speaker EMITTED_TYPES: 0x01, 0x03 // list which types you emit COUPLING_COMPLIANCE: Full-v0.1 // all 3 types, listen + emit

Reference Implementation

XOceanus is the reference implementation of this protocol. All 77 engines are Level 3 compliant for the three public types. The C++ contract lives in Source/Core/SynthEngine.h:

// Speaker interface — emit coupling signal virtual float getSampleForCoupling(int channel, int sampleIndex) const = 0; // Listener interface — receive coupling modulation virtual void applyCouplingInput(CouplingType type, float amount, const float* sourceBuffer, int numSamples) = 0;

XOceanus implements 15 coupling types: the 3 public types defined here plus 12 extended types. The three public types map to underlying XOceanus mechanisms as follows:

Public TypeXOceanus MechanismDescription
0x01 SympatheticPitchToPitch + FilterToFilterApplied bilaterally
0x02 ComplementaryAmpToChoke + RhythmToBlendApplied inversely
0x03 ResponsiveAmpToFilter + EnvToMorphApplied unilaterally

This mapping is informational. Third-party implementations are not required to use these mechanisms — the protocol specifies the semantic contract, not the implementation strategy.

Extension Mechanism

The protocol is designed to grow. Type IDs are allocated from reserved ranges; future public types do not break existing compliant implementations.

RangeDesignation
0x01 – 0x03Public v0.1 types (this spec)
0x04 – 0x7FReserved for future public spec versions
0x80 – 0xEFAvailable for private / extended implementations
0xF0 – 0xFFExperimental / vendor-specific (no compatibility guarantees)

Future packet extensions must preserve the 16-byte v0.1 header. Additional fields are appended after reserved. Receivers should process the first 16 bytes and ignore the remainder when a packet is larger than expected.

FAQ

Is this a replacement for MIDI?
No. MIDI describes what notes to play and what control values to send. Coupling describes relationships between simultaneously-playing voices. They operate at different semantic levels and are complementary.
Does the source engine need to know who is listening?
No. The source engine exposes a coupling output and has no knowledge of which engines, if any, are receiving it. The host matrix manages routing. Engines are relationship-capable, not relationship-aware.
What happens if a target engine doesn’t support a received type?
The target should silently ignore packets with unsupported type IDs. It must not crash or produce undefined behavior. No response is a valid and correct response.
Can coupling routes be bidirectional?
Yes — via two separate routes, one from A to B and one from B to A, each with its own type and amount. The XOceanus KnotTopology extended type implements a special case where bidirectionality is architecturally enforced: neither engine can be decoupled without destroying both routes.
How does Complementary coupling differ from sidechain compression?
Sidechain compression acts on the mixed output signal through gain reduction. Complementary coupling acts on the target engine’s parameter space before audio is generated — it influences how the engine synthesizes sound, not how loud the result is. The timbral character of the receding engine changes, not just its volume.
Is this open source?
The protocol specification is published under CC BY 4.0. The reference implementation (XOceanus) is open source under its own repository license. Third parties may implement the protocol without restriction.
Can amount be negative?
The v0.1 spec constrains amount to [0.0, 1.0]. Extended implementations may use the signed float range [-1.0, 1.0] — negative amounts typically invert the coupling polarity (Sympathetic becomes dissonant instead of reinforcing). This is not part of v0.1 compliance.
