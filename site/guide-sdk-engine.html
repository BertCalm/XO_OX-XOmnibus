SDK Guide — Architecture #14

How to Build an XOceanus Engine in 200 Lines

Every engine in XOceanus — all 76 of them — is a standalone DSP module that implements one C++ interface. The SDK removes JUCE as a dependency at the boundary. You can write a new engine with a text editor, a C++17 compiler, and this tutorial.

Read Time25 min
Words~5,000
AccentXO Gold #E9C46A

Every engine in XOceanus implements the same interface. ODDFELIX, with its snapshot synthesis and neon-tetra personality, implements it. ORGANISM, with its Wolfram automaton driving 16 cells of DSP, implements it. OPERA, with Kuramoto oscillator networks and an autonomous dramatic conductor, implements it. The interface is short. The surface area a third-party developer needs to understand is small. This tutorial builds a complete working engine — one sawtooth oscillator, one one-pole lowpass filter, four parameters, one macro — and integrates it into XOceanus in about 200 lines.

The XOceanus SDK lives in SDK/include/xoceanus/. It has three headers: SynthEngine.h (the interface), CouplingTypes.h (the cross-engine modulation enum), and EngineModule.h (the export macro for compiled shared libraries). None of them include JUCE. JUCE types — juce::AudioBuffer, juce::MidiBuffer, juce::String — are replaced at the boundary with plain C++ equivalents: StereoBuffer, MidiEventList, std::string. The adapter layer inside XOceanus translates between them. If you are building an engine that will ship as a compiled module, you never touch JUCE at all.

If you are building inside the XOceanus source tree — contributing to the main repo or adding a private engine to your fork — you implement the JUCE-coupled Source/Core/SynthEngine.h interface instead. The structure is identical. The difference is only at the type boundary. This tutorial uses the SDK interface throughout, but all the patterns translate directly.

Section 1: The Interface

The full xoceanus::SynthEngine class is in SDK/include/xoceanus/SynthEngine.h. Here are the pure-virtual methods every engine must implement, each with its contract:

SDK/include/xoceanus/SynthEngine.h — required interface 
namespace xoceanus {

class SynthEngine {
public:
    virtual ~SynthEngine() = default;

    // ── Lifecycle ────────────────────────────────────────────────────────

    // Allocate buffers, initialize state. Called once before audio starts.
    // Memory allocation IS allowed here. Never in renderBlock().
    virtual void prepare(double sampleRate, int maxBlockSize) = 0;

    // Free non-essential resources when audio stops.
    virtual void releaseResources() {}

    // Reset oscillator phases, filter state, envelopes — without reallocating.
    // Called on preset change and transport reset.
    virtual void reset() = 0;

    // ── Audio ────────────────────────────────────────────────────────────

    // Render a block of audio. Process MIDI. Real-time safe: no allocation,
    // no blocking, no exceptions. Write into buffer additively.
    virtual void renderBlock(StereoBuffer& buffer, const MidiEventList& midi) = 0;

    // ── Coupling ─────────────────────────────────────────────────────────

    // Return a cached output sample for coupling reads. O(1) — just read
    // from a pre-filled member array. channel: 0=left, 1=right, 2=env level.
    virtual float getSampleForCoupling(int channel, int sampleIndex) const
        { return 0.0f; } // default: no coupling output

    // Receive modulation from another engine. Accumulate; consume in renderBlock().
    virtual void applyCouplingInput(CouplingType type, float amount,
                                     const float* sourceBuffer, int numSamples) {}

    // ── Parameters ───────────────────────────────────────────────────────

    // Return all parameter definitions. Called once at startup.
    virtual std::vector<ParameterDef> getParameterDefs() const = 0;

    // Set / get a parameter by ID. Called from non-audio thread.
    virtual void setParameter(const std::string& id, float value) {}
    virtual float getParameter(const std::string& id) const { return 0.0f; }

    // ── Identity ─────────────────────────────────────────────────────────

    // Unique identifier — use an O-word (e.g. "Minimal", "Oscillum").
    virtual std::string getEngineId() const = 0;

    // Accent colour for UI theming. Return an RGBA struct.
    virtual Colour getAccentColour() const { return { 128, 128, 128 }; }

    virtual int getMaxVoices() const { return 1; }
    virtual int getActiveVoiceCount() const { return 0; }
};

} // namespace xoceanus

Six pure virtuals are mandatory: prepare(), reset(), renderBlock(), getParameterDefs(), and getEngineId(). Everything else has a default implementation you can override when you need it. Coupling output, coupling input, and parameter get/set are all optional until you want them.

Section 2: Building MinimalEngine Step by Step

MinimalEngine has one sawtooth oscillator, a one-pole lowpass filter, an amplitude envelope, and four parameters. It uses the min_ parameter prefix. Each step below adds one capability, building to a complete working engine.

Step 1 — The Shell

Start with the class declaration and identity methods. Pick an O-word engine ID and a parameter prefix you will never change.

Step 1 — class declaration and identity 
#pragma once
#include <xoceanus/SynthEngine.h>
#include <cmath>
#include <atomic>

class MinimalEngine : public xoceanus::SynthEngine {
public:
    std::string getEngineId() const override { return "Minimal"; }

    xoceanus::Colour getAccentColour() const override
    {
        // XO Gold — use your own accent colour here
        return { 0xE9, 0xC4, 0x6A };
    }

    int getMaxVoices() const override { return 1; }
    int getActiveVoiceCount() const override { return gate_ ? 1 : 0; }

private:
    // DSP state — all initialized in prepare() / reset()
    double sampleRate_ = 48000.0;
    float  phase_  = 0.0f;   // oscillator phase [0, 1)
    float  freq_   = 440.0f; // current note frequency
    float  vel_    = 0.0f;   // note velocity [0, 1]
    bool   gate_   = false;
    float  filterState_ = 0.0f; // one-pole filter memory
    float  ampEnv_      = 0.0f; // amplitude envelope follower

    // Param values — written by setParameter(), read in renderBlock()
    std::atomic<float> pCutoff_  { 0.5f }; // min_cutoff [0,1]
    std::atomic<float> pReso_    { 0.0f }; // min_reso   [0,1]
    std::atomic<float> pAttack_  { 0.01f}; // min_attack seconds
    std::atomic<float> pRelease_ { 0.3f }; // min_release seconds
};

Step 2 — Parameters

getParameterDefs() declares all parameters with their ID, display name, range, default, and skew. IDs must follow the {prefix}_{name} convention and are frozen after the first release — changing them breaks saved patches.

Step 2 — parameter declarations 
std::vector<xoceanus::ParameterDef> getParameterDefs() const override
{
    // ParameterDef: { id, name, min, max, default, step, skew }
    // skew < 1.0 = log-like curve (good for frequency/time params)
    return {
        { "min_cutoff",  "Filter Cutoff",  0.0f, 1.0f, 0.5f,  0.001f, 0.4f },
        { "min_reso",    "Resonance",      0.0f, 1.0f, 0.0f,  0.001f, 1.0f },
        { "min_attack",  "Attack",         0.001f,4.0f, 0.01f, 0.001f, 0.35f},
        { "min_release", "Release",        0.01f, 8.0f, 0.3f,  0.001f, 0.35f},
    };
}

// Called from non-audio thread when host automates a parameter.
void setParameter(const std::string& id, float v) override
{
    if      (id == "min_cutoff")  pCutoff_.store(v);
    else if (id == "min_reso")    pReso_.store(v);
    else if (id == "min_attack")  pAttack_.store(v);
    else if (id == "min_release") pRelease_.store(v);
}

float getParameter(const std::string& id) const override
{
    if      (id == "min_cutoff")  return pCutoff_.load();
    else if (id == "min_reso")    return pReso_.load();
    else if (id == "min_attack")  return pAttack_.load();
    else if (id == "min_release") return pRelease_.load();
    return 0.0f;
}
Doctrine D004 — Dead Parameters Are Broken Promises

Every parameter you declare in getParameterDefs() must affect audio output. An engine with a declared parameter that does nothing fails D004. The seance will find it. Your users will notice it. Audit your parameter list before submitting.

Step 3 — Prepare

prepare() is called once before audio starts. It is the only place allocation is permitted. Store the sample rate and clear all DSP state. The sample rate comes from the host — never hardcode 44100 or 48000.

Step 3 — prepare() and reset() 
void prepare(double sampleRate, int /*maxBlockSize*/) override
{
    sampleRate_ = sampleRate;
    reset();
    // Allocate any per-block scratch buffers here if needed.
    // This engine is simple enough to need none.
}

void reset() override
{
    phase_       = 0.0f;
    filterState_ = 0.0f;
    ampEnv_      = 0.0f;
    gate_        = false;
    vel_         = 0.0f;
}

Step 4 — Note Handling

MIDI events arrive in the MidiEventList parameter of renderBlock(). The MidiEvent struct provides helpers: isNoteOn(), isNoteOff(), getNoteNumber(), getFloatVelocity(). Parse them at the correct sample offset within the block.

Step 4 — MIDI-to-frequency conversion and note gate 
// Convert MIDI note number to Hz using equal temperament.
// Standard formula: 440 Hz * 2^((note - 69) / 12)
static float midiToHz(int note) noexcept
{
    return 440.0f * std::pow(2.0f, (note - 69) / 12.0f);
}

// Call this per-event before the DSP loop
void handleEvent(const xoceanus::MidiEvent& e)
{
    if (e.isNoteOn()) {
        freq_ = midiToHz(e.getNoteNumber());
        vel_  = e.getFloatVelocity();
        gate_ = true;
        // Don't reset phase — legato re-trigger sounds better
    }
    else if (e.isNoteOff()) {
        gate_ = false;
    }
}

Step 5 — The Render Loop

renderBlock() is called on the audio thread at the host's buffer rate. It must complete within the buffer duration. Read all parameter atomics once at the top of the function — the ParamSnapshot pattern — then use the captured values per-sample. This avoids data races and costs nothing.

Step 5 — renderBlock() with ParamSnapshot pattern 
void renderBlock(xoceanus::StereoBuffer& buf, const xoceanus::MidiEventList& midi) override
{
    // ── ParamSnapshot: read atomics once per block ─────────────────────
    const float cutoff  = pCutoff_.load();
    const float attack  = pAttack_.load();
    const float release = pRelease_.load();

    // One-pole filter coefficient from normalised cutoff [0,1]
    // Map to 20 Hz – 18 kHz, then compute matched-Z coefficient:
    // coeff = exp(-2π * fc / sr)   — the correct IIR formula
    const float fc    = 20.0f * std::pow(900.0f, cutoff);
    const float coeff = std::exp(-2.0f * 3.14159265f * fc / (float)sampleRate_);

    // Amplitude envelope time constants in samples
    const float attackCoeff  = std::exp(-1.0f / (attack  * (float)sampleRate_));
    const float releaseCoeff = std::exp(-1.0f / (release * (float)sampleRate_));

    const float phaseInc = freq_ / (float)sampleRate_;

    int eventIdx = 0;

    for (int i = 0; i < buf.numSamples; ++i)
    {
        // ── MIDI events at their correct sample offset ─────────────────
        while (eventIdx < midi.numEvents && midi.events[eventIdx].sampleOffset == i)
            handleEvent(midi.events[eventIdx++]);

        // ── Sawtooth oscillator ────────────────────────────────────────
        const float saw = phase_ * 2.0f - 1.0f; // [-1, +1]
        phase_ += phaseInc;
        if (phase_ >= 1.0f) phase_ -= 1.0f;

        // ── One-pole lowpass ───────────────────────────────────────────
        filterState_ = coeff * filterState_ + (1.0f - coeff) * saw;

        // ── Amplitude envelope (gate-driven exponential) ───────────────
        const float envTarget = gate_ ? vel_ : 0.0f;
        const float envCoeff  = (envTarget > ampEnv_) ? attackCoeff : releaseCoeff;
        ampEnv_ = envCoeff * ampEnv_ + (1.0f - envCoeff) * envTarget;

        // ── Write output (additive — host may have mixed other engines) ─
        const float out = filterState_ * ampEnv_;
        buf.left [i] += out;
        buf.right[i] += out;
    }

    // Denormal protection: flush subnormals that accumulate in filter memory
    if (std::abs(filterState_) < 1e-18f) filterState_ = 0.0f;
    if (std::abs(ampEnv_)      < 1e-18f) ampEnv_      = 0.0f;
}
Critical: use matched-Z IIR coefficients

The correct one-pole coefficient is exp(-2πfc/sr). The Euler approximation w/(w+1) is wrong above 1 kHz and produces audible frequency error. Always derive the sample rate from the sampleRate argument to prepare() — never hardcode 44100 or 48000. Users with 96 kHz interfaces will hear broken filters otherwise.

The Complete 200-Line Engine

The six steps above assemble into a single header file. Here is the complete MinimalEngine.h:

MinimalEngine.h — complete implementation (~200 lines) 
#pragma once
#include <xoceanus/SynthEngine.h>
#include <cmath>
#include <atomic>
#include <string>
#include <vector>

class MinimalEngine : public xoceanus::SynthEngine {
public:

    // ── Identity ─────────────────────────────────────────────────────────
    std::string getEngineId() const override { return "Minimal"; }
    xoceanus::Colour getAccentColour() const override { return {0xE9,0xC4,0x6A}; }
    int getMaxVoices() const override { return 1; }
    int getActiveVoiceCount() const override { return gate_ ? 1 : 0; }

    // ── Parameters ───────────────────────────────────────────────────────
    std::vector<xoceanus::ParameterDef> getParameterDefs() const override
    {
        return {
            { "min_cutoff",  "Filter Cutoff",  0.0f,  1.0f,  0.5f,  0.001f, 0.4f  },
            { "min_reso",    "Resonance",      0.0f,  1.0f,  0.0f,  0.001f, 1.0f  },
            { "min_attack",  "Attack",         0.001f,4.0f,  0.01f, 0.001f, 0.35f },
            { "min_release", "Release",        0.01f, 8.0f,  0.3f,  0.001f, 0.35f },
        };
    }

    void setParameter(const std::string& id, float v) override
    {
        if      (id == "min_cutoff")  pCutoff_.store(v);
        else if (id == "min_reso")    pReso_.store(v);
        else if (id == "min_attack")  pAttack_.store(v);
        else if (id == "min_release") pRelease_.store(v);
    }

    float getParameter(const std::string& id) const override
    {
        if      (id == "min_cutoff")  return pCutoff_.load();
        else if (id == "min_reso")    return pReso_.load();
        else if (id == "min_attack")  return pAttack_.load();
        else if (id == "min_release") return pRelease_.load();
        return 0.0f;
    }

    // ── Lifecycle ────────────────────────────────────────────────────────
    void prepare(double sampleRate, int) override
    {
        sampleRate_ = sampleRate;
        reset();
    }

    void reset() override
    {
        phase_ = filterState_ = ampEnv_ = 0.0f;
        gate_  = false;
        vel_   = 0.0f;
    }

    // ── Audio ────────────────────────────────────────────────────────────
    void renderBlock(xoceanus::StereoBuffer& buf,
                     const xoceanus::MidiEventList& midi) override
    {
        // ParamSnapshot — read atomics once, use captured values per sample
        const float cutoff  = pCutoff_.load();
        const float attack  = pAttack_.load();
        const float release = pRelease_.load();
        const float sr      = (float)sampleRate_;

        // One-pole lowpass: matched-Z coefficient
        const float fc    = 20.0f * std::pow(900.0f, cutoff);
        const float coeff = std::exp(-2.0f * 3.14159265f * fc / sr);

        // Envelope time constants in samples
        const float aCoeff = std::exp(-1.0f / (attack  * sr));
        const float rCoeff = std::exp(-1.0f / (release * sr));
        const float phInc  = freq_ / sr;

        int evIdx = 0;

        for (int i = 0; i < buf.numSamples; ++i)
        {
            while (evIdx < midi.numEvents && midi.events[evIdx].sampleOffset == i)
                handleEvent(midi.events[evIdx++]);

            // Sawtooth: phase sweeps [0, 1) → sample in [-1, +1]
            const float saw = phase_ * 2.0f - 1.0f;
            phase_ += phInc;
            if (phase_ >= 1.0f) phase_ -= 1.0f;

            // One-pole lowpass (D001: velocity shapes filter — add vel_ offset)
            filterState_ = coeff * filterState_ + (1.0f - coeff) * saw;

            // Amplitude envelope
            const float envTarget = gate_ ? vel_ : 0.0f;
            const float envC = (envTarget > ampEnv_) ? aCoeff : rCoeff;
            ampEnv_ = envC * ampEnv_ + (1.0f - envC) * envTarget;

            const float out = filterState_ * ampEnv_;
            buf.left [i] += out;
            buf.right[i] += out;
        }

        // Denormal protection
        if (std::abs(filterState_) < 1e-18f) filterState_ = 0.0f;
        if (std::abs(ampEnv_)      < 1e-18f) ampEnv_      = 0.0f;
    }

private:
    double sampleRate_ = 48000.0;
    float  phase_  = 0.0f;
    float  freq_   = 440.0f;
    float  vel_    = 0.0f;
    bool   gate_   = false;
    float  filterState_ = 0.0f;
    float  ampEnv_      = 0.0f;

    std::atomic<float> pCutoff_  { 0.5f  };
    std::atomic<float> pReso_    { 0.0f  };
    std::atomic<float> pAttack_  { 0.01f };
    std::atomic<float> pRelease_ { 0.3f  };

    void handleEvent(const xoceanus::MidiEvent& e) noexcept
    {
        if      (e.isNoteOn())  { freq_ = midiToHz(e.getNoteNumber());
                                   vel_  = e.getFloatVelocity(); gate_ = true; }
        else if (e.isNoteOff()) { gate_ = false; }
    }

    static float midiToHz(int note) noexcept
    { return 440.0f * std::pow(2.0f, (note - 69) / 12.0f); }
};

Section 3: The XOCEANUS_EXPORT_ENGINE Macro

When you ship an engine as a compiled shared library (.dylib / .so / .dll), one macro in your .cpp file exports the two C functions XOceanus expects at the module boundary: xoceanus_create_engine(), which returns a new instance, and xoceanus_engine_info(), which fills metadata without instantiating the engine.

MinimalEngine.cpp — the only .cpp file you need 
#include "MinimalEngine.h"
#include <xoceanus/EngineModule.h>

// Arguments: ClassName, ID, DisplayName, Prefix, R, G, B, Version, Author
XOCEANUS_EXPORT_ENGINE(
    MinimalEngine,         // your C++ class
    "Minimal",             // engine ID — use an O-word for fleet engines
    "Minimal Engine",      // display name shown in UI
    "min_",               // parameter prefix — FROZEN after first release
    0xE9, 0xC4, 0x6A,       // accent colour RGB (XO Gold here)
    "1.0.0",               // semver string
    "Your Name"            // author / studio
)

That is the entire public API surface. XOceanus loads the shared library, calls xoceanus_engine_info() to read metadata without allocating anything, then calls xoceanus_create_engine() to get a live instance. The EngineMetadata struct also carries the SDK version integer — if a future XOceanus version introduces a breaking ABI change, the version field allows the host to reject incompatible modules gracefully.

Section 4: Integration Checklist

Whether you are building inside the XOceanus source tree or shipping a compiled module, the integration path is the same four steps:

Integration Steps
  • Copy MinimalEngine.h to Source/Engines/Minimal/MinimalEngine.h
  • Add #include "Engines/Minimal/MinimalEngine.h" to Source/Core/EngineRegistry.h and register the engine in the registry constructor
  • Add a row to the engine table in CLAUDE.md and Docs/xoceanus_master_specification.md (engine ID, parameter prefix, accent colour)
  • Copy any preset .xometa files to Presets/XOceanus/{mood}/
  • Run cmake --build build and verify zero errors
  • Run auval -v aumu XOmn XOOX and verify PASS
  • Run the engine health check skill (/engine-health-check) against the new engine

For a compiled third-party module, skip the registry step and drop the compiled .xomodule into the XOceanus modules folder (~/Library/Application Support/XO_OX/Modules/ on macOS). XOceanus scans this folder at startup, loads every valid module, and registers its engines automatically.

Section 5: The Six Doctrines — Minimum Viable

Every engine in the fleet is evaluated against six doctrines. Here is what minimum viable compliance looks like for a new engine. Meeting these thresholds is the baseline for integration. The seance process evaluates whether you have met the spirit, not just the letter.

ID Doctrine Minimum Viable Implementation
D001 Velocity Must Shape Timbre Velocity should scale filter cutoff, not just amplitude. In MinimalEngine, add vel_ * 0.3f to the cutoff before computing fc. Higher velocity = brighter sound.
D002 Modulation is the Lifeblood At minimum: 2 LFOs with rate and depth parameters, mod wheel mapped to filter or pitch, 4 working macros, at least 4 mod matrix slots. Static engines fail this doctrine.
D003 The Physics IS the Synthesis If your engine models a physical system (string, membrane, room), cite the paper in a source comment. No citation requirement for purely abstract synthesis.
D004 Dead Parameters Are Broken Promises Run a manual check: zero every parameter that is not the one you are testing. Verify each parameter produces audible change in isolation. No declared parameter may be a no-op.
D005 An Engine That Cannot Breathe Is a Photograph At least one LFO must have a rate floor at or below 0.01 Hz (one full cycle per 100 seconds). This allows the engine to self-modulate over very long timescales.
D006 Expression Input Is Not Optional Velocity must modulate timbre (D001 above). At least one additional expression input: aftertouch, mod wheel CC1, or expression pedal CC11. Engines without any CC response fail this doctrine.

MinimalEngine as shown above passes D004 (all four params wired), partially satisfies D001 (velocity modulates amplitude; adding cutoff offset fully satisfies it), and fails D002, D005, and D006 out of the box. That is expected — MinimalEngine is a scaffold, not a production engine. A fleet engine needs at least 2 LFOs before it ships.

Section 6: Going Further

Adding Coupling

To accept modulation from other engines, override applyCouplingInput(). The CouplingType enum in SDK/include/xoceanus/CouplingTypes.h has 15 types. The most common for a filter-and-oscillator engine are AmpToFilter (another engine's amplitude modulates your cutoff) and LFOToPitch (another engine's LFO modulates your pitch). Accumulate the coupling signal in a member variable and consume it during renderBlock():

Accepting coupling input 
// Member: float couplingFilterMod_ = 0.0f;

void applyCouplingInput(xoceanus::CouplingType type, float amount,
                         const float*, int) override
{
    if (type == xoceanus::CouplingType::AmpToFilter)
        couplingFilterMod_ = amount; // added to cutoff in renderBlock()
}

// Then in the renderBlock() ParamSnapshot section:
const float cutoff = pCutoff_.load() + couplingFilterMod_;
couplingFilterMod_ = 0.0f; // reset each block

To send coupling output (allowing other engines to read your audio), populate a small cache array during renderBlock() and return values from getSampleForCoupling(). The array only needs to hold one block — typically 64 to 2048 samples depending on host buffer size.

The 6D Sonic DNA System

Every preset in XOceanus carries a 6D Sonic DNA fingerprint: brightness, warmth, movement, density, space, and aggression — each on a 0–1 scale. When you write presets for your engine, fill in the "dna" field in the .xometa JSON. The preset browser uses these values for filtering and recommendation. An engine with no DNA values will appear in search results but will not surface through the mood browser. A preset JSON with DNA looks like this:

Shared DSP Utilities

The Source/DSP/ directory contains six shared utilities extracted from the fleet that cover the most common engine requirements: StandardLFO.h (5-shape LFO with D005 rate floor), FilterEnvelope.h (general-purpose ADSR), PitchBendUtil.h (MIDI pitch wheel to frequency ratio), VoiceAllocator.h (LRU voice stealing), GlideProcessor.h (frequency-domain portamento), and ParameterSmoother.h (one-pole zipper-noise-free parameter smoothing). Before writing your own LFO or ADSR, check whether one of these covers your use case. Engines built inside the source tree can include them directly. Compiled SDK modules should copy the relevant utility header into their own source.

The goal of the interface

XOceanus has 76 engines because each one started as an independent instrument. The interface is short by design. The constraint that matters is not the API surface — it is the doctrines. An engine that passes D001–D006 can be trusted to sound like it has a point of view. That is what makes a character instrument rather than a feature matrix.
